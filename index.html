プログラミング教育支援システム

hibikicode（ひびきコード）は、テキストベースのプログラミング学習において、
学習者一人ひとりの特性を可視化し、最適な指導を可能にする「教育ケア」プラットフォームです。
視覚的に楽しめる学習環境と、独自の学習者分析機能 「hibikicode feedback」により、
講師の経験値に依存しない質の高い個別指導を目指します。

"Experience × Data で、学習者の『真の力』を引き出す。"

いま、プログラミング教育の現場で起きていること

課題① 学習者への「ケアの欠如」

既存の教材は単調で、「英語が苦手」「将来像がイメージできない」といった理由から、 学習意欲を失ってしまう学習者が少なくありません。
それでも「なんとなく続けている」生徒は、教室内で静かに埋もれていきます。

課題② 講師側の「分析する余裕の欠如」

一対多数の指導では、講師が全員の躓きや潜在能力を 客観的に分析する余裕がほとんどありません。
質問できない学習者や、独自の才能を持ちながらも 画一的な指導に埋もれてしまう学習者が生まれています。

結果として、指導の質は講師個人の勘や経験に大きく依存してしまいます。
hibikicodeは、この状況をテクノロジーの力で変え、 学習者の個性に寄り添う「教育ケア」を標準装備にすることを目指します。

多様な学習環境で「ワクワク」を届ける

テキストベース学習の入り口

Python を中心としたテキストベースの環境で、
「コードを書いて・動かして・結果を見る」流れをやさしく体験。
英語への苦手意識をやわらげる日本語 UI と解説も用意します。

図形描画・デザイン×プログラミング

図形描画やアニメーション、p5.js 的な表現を Python で体験。
「目に見える作品づくり」を通じて、退屈しない学習プロセスを設計します。

画像処理で数学とつながる

行列・座標・画素といった概念を、画像処理の実験を通して学習。
「なんのための数学か？」が、手元のコードと画面の変化で分かります。

Core Feature

学習者分析「hibikicode feedback」

無意識の行動ログをデータに変え、
学習者の「長所」と「課題」を可視化する分析エンジン。

1

特徴量抽出

入力ログ、課題達成度、変数命名の傾向、
タイピング速度など、学習者の行動を データとして収集します。

2

次元削減

多次元の特徴量を 2次元空間へマッピング。
学習者同士の「近さ・違い」を 直感的に把握できます。

3

クラスタリング

似た特徴を持つ学習者同士をグループ化。
「同じような躓き方をする生徒」や 「似た強みを持つ生徒」を発見します。

LLM

センスメイキング

「創造性豊かだが継続力が弱い」
「理解するよりも暗記が得意」
「英語力が強い」「変数が使いこなせていない」など、
LLM が自然言語で意味付けし、 講師に分かりやすく提示します。

UMAP learner space（イメージ）

Cluster A: 創造性高めタイプ

創造性が高く、アイデア発想や表現力に優れる一方で、 課題の継続や基礎文法の反復がやや弱い傾向があります。
小さな成功体験を積み重ねる短時間課題がおすすめです。

Cluster B: 覚えて進むタイプ

理解よりもパターン暗記が得意で、サンプルコードの写経は速いが、 応用問題でつまずきやすい傾向があります。
日本語での概念説明と、少しずつ条件を変える練習課題が有効です。

Cluster C: 英語強みタイプ

英単語やドキュメント読解に強みがあり、 英語のエラーメッセージへの耐性も高いグループです。
公式ドキュメントや英語教材を活かした発展課題に誘導できます。

これらのクラスタ情報は、「今この瞬間、どんな声掛けが最適か？」を判断するための コンパスとして、講師に提示されます。

現場での 4 年間の葛藤から生まれたプラットフォーム

4年間にわたるプログラミング教室での講師経験のなかで、
目の前で退屈そうにしている学習者や、 英語の壁に阻まれて自信を失っていく姿を何度も見てきました。

また、新人講師が生徒との距離感や声掛けに戸惑い、 「本当はもっと力を伸ばしてあげられるはずなのに」と感じる場面も多くありました。

一対多数の環境では、全員の深い躓きに気づくことは簡単ではありません。
その無力感から生まれたのが、
「講師の勘や経験だけに頼らず、データに基づいて学習者の個性を守りたい」 という想いでした。

hibikicode は、学習者と指導者の両方を支える「教育ケア」の基盤として、
現場の経験とデータサイエンスをつなぐことを目指しています。
